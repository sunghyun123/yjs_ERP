# 전기공사 ERP 자체 구축 설계 문서

> 작성일: 2026-06-04  
> 목적: 기존 EMAX ERP(Windows 데스크탑 앱)를 웹 기반으로 자체 구축  
> 작성 기준: 실제 ERP 화면, 엑셀 데이터(투입실적현황.xlsx, 수주대장조회.xlsx, 매출손익현황.xlsx) 분석 결과

## 1. 프로젝트 배경

### 기존 ERP 문제점
- Windows 데스크탑 앱 (EMAX ERP) — PC에서만 접근 가능, 모바일 불가
- 자동화 API 전무 — 투입실적 API 하나 요청했으나 2개월째 미완성
- UI/UX 낙후 (2000년대 초반 WinForms 스타일)
- 소수점 계산 버그 존재
- 외주 개발사 의존으로 추가 기능 개발 지연/비용 발생

### 자체 구축 목표
- 웹 기반 (모바일 접근 가능)
- DB 생성 즉시 REST API 자동 제공 (Supabase)
- 기존 ERP의 3개 핵심 출력물 완전 재현
  - 수주대장 조회
  - 투입실적 현황
  - 매출손익 현황

### 입력 부담 감소에 대한 현실적 판단

**현재 투입실적 입력 흐름:**
```
현장팀 시공팀작업일지 수기 작성
→ 사진 촬영 후 카카오톡 전송
→ 공무팀이 사진을 보며 ERP에 수동 타이핑 (건당 약 10초 이내)
```

**시공팀작업일지 수기 작성은 경영방침상 반드시 유지해야 하는 과정이다.**  
현장팀의 디지털 전환은 현재 시점에서 불가능하며, 변경 대상이 아님.

**AI 사진 분석(OCR) 방식은 현 시점에서 배제:**
- 정확도 문제: 99% 정확해도 오류 검증에 추가 시간 발생
- 속도 문제: AI 분석 30초 > 공무팀 수동 입력 10초
- 비용 문제: 공사 1건당 사진 1만 장 이상인 경우 존재 (AI API 비용 과다)
- 추후 AI 속도/비용이 개선되면 재검토 가능

**따라서 자체 구축의 실질적 가치는 입력 방식 혁신이 아니라:**
1. PC 의존 제거 → 어디서든 웹 접근
2. API 즉시 확보 → 자동화 연동 가능
3. 계산 로직 직접 관리 → 버그 즉시 수정
4. 개발사 의존 제거 → 기능 추가/수정 자유

**공무팀 UX 개선은 거창한 자동화가 아닌 실용적 편의 기능으로:**
- 공사명 자동완성 (지중No 입력 시 공사명 자동 표시)
- 전날 복사 버튼 (동일 현장 연속 공사 시 인원 그대로 복사)
- 키보드 Tab 이동 최적화 (마우스 없이 빠른 입력)

## 2. 기술 스택

> **설계 원칙:** 이 ERP는 회사 전체의 재무 자산을 다룬다.  
> 화려함보다 **안정성, 보안성, 유지보수성**을 우선한다.  
> 개발의 99%를 AI에 위임하는 구조이므로, AI가 잘 생성하고 사람이 검토하기 쉬운 표준 패턴을 고수한다.

| 역할 | 도구 |
|------|------|
| DB + REST API | Supabase PostgreSQL (무료 플랜) |
| 프론트엔드 | Next.js 14 (App Router) + **TypeScript** |
| UI 컴포넌트 | shadcn/ui + Tailwind CSS |
| 입력 검증 | **Zod** |
| 차트 | Recharts |
| 배포 | **기존 VPS** (hpa_1.4-g2 / 1vCPU / 4GB RAM) |
| 웹서버 | Nginx (리버스 프록시 + SSL) |
| 프로세스 관리 | PM2 |

### 2-1. Supabase

**선택 이유:**  
이 프로젝트의 핵심 요구사항이 "API 즉시 확보"다. Supabase는 PostgreSQL 테이블을 만드는 순간 REST API와 실시간 구독이 자동으로 생성된다. 기존 ERP 개발사에 API 하나 요청해서 2개월 기다린 문제가 처음부터 존재하지 않는다.

**장점:**
- 테이블 생성 즉시 REST API 자동 생성 (`/rest/v1/테이블명`)
- PostgreSQL 기반 → 관계형 데이터, 복잡한 집계 쿼리 모두 가능
- Auth 내장 → 로그인/권한 별도 구현 불필요
- 대시보드에서 DB를 엑셀처럼 직접 조회/수정 가능 → 비개발자도 데이터 확인 가능
- 오픈소스 → 나중에 자체 서버로 이전 가능 (vendor lock-in 최소)
- 무료 플랜으로 이 규모(수주 500건, 투입실적 수천 건)는 충분히 커버

**단점/주의사항:**
- 무료 플랜은 2주 미접속 시 프로젝트 일시 정지 → 유료 플랜($25/월) 전환 필요
- Supabase 자체가 외부 서비스이므로 장애 시 접근 불가 (SLA 99.9%)
- 복잡한 트랜잭션 로직은 DB 함수(PostgreSQL Function)로 작성해야 함

**대안으로 고려했던 것:**
- `Firebase`: NoSQL 구조라 수주-투입실적-기성 같은 관계형 데이터에 부적합
- `직접 PostgreSQL + Express API 서버`: 가능하지만 API 서버 직접 구축/관리 부담
- `PocketBase`: 더 단순하지만 생태계가 작고 복잡한 집계 쿼리에 한계

### 2-2. Next.js 14 (App Router)

**선택 이유:**  
1인 개발자가 AI(Claude, Cursor)와 함께 개발할 때 Next.js로 생성되는 코드 품질이 가장 높다. 프론트엔드와 백엔드(API Route)를 하나의 프로젝트에서 관리할 수 있어 구조가 단순해진다.

**장점:**
- AI 코드 생성 최적화 — Claude/Cursor가 Next.js+shadcn 조합으로 코드를 가장 잘 만들어줌
- 풀스택 단일 프로젝트 — 프론트/백엔드 레포 분리 없이 하나로 관리
- Server Components → 민감한 DB 쿼리를 서버에서 처리, 클라이언트에 키 노출 없음
- Vercel 무료 배포와 네이티브 연동

**단점/주의사항:**
- App Router 학습 곡선 — `use client` / `use server` 개념 처음엔 헷갈림
- React Server Components는 상태(useState) 사용 불가 → 화면 설계 시 구분 필요
- 순수 SPA보다 빌드/배포 시간이 약간 길다

**대안으로 고려했던 것:**
- `React SPA + 별도 Express 서버`: 구조 분리로 관리 포인트 2배
- `Vue.js / Nuxt`: 가능하지만 AI 코드 생성 품질이 Next.js 대비 낮음
- `Django + Jinja 템플릿`: Python 익숙하면 빠르지만 현대적 UI 구현에 제약

### 2-3. shadcn/ui + Tailwind CSS

**선택 이유:**  
1인 개발자의 최대 약점인 "디자인과 UI 컴포넌트"를 해결한다. shadcn/ui는 테이블, 폼, 모달, 드롭다운 등 ERP에 필요한 컴포넌트가 이미 다 있고, AI가 이 조합으로 코드를 가장 잘 생성해준다.

**장점:**
- 컴포넌트를 프로젝트 안에 복사해서 쓰는 구조 → 의존성 없음, 자유롭게 수정 가능
- DataTable, Form, Select, Dialog 등 ERP 핵심 컴포넌트 기성품 제공
- Tailwind → 클래스명으로 스타일 지정, CSS 파일 별도 관리 불필요
- AI가 shadcn+Tailwind 조합 코드를 매우 잘 생성함

**단점/주의사항:**
- Tailwind 클래스명을 처음엔 외워야 함 (금방 익숙해짐)
- 컴포넌트가 프로젝트 내부에 들어오기 때문에 파일 수가 많아짐
- 디자인 커스터마이징이 자유로운 만큼 일관성 유지를 직접 신경 써야 함

**대안으로 고려했던 것:**
- `Ant Design`: 컴포넌트 품질 좋지만 무겁고 커스터마이징 어려움
- `Material UI`: 구글 스타일로 고정되어 있어 한국 업무 시스템 느낌과 맞지 않음
- `Bootstrap`: 오래됐고 React 친화적이지 않음

### 2-4. Recharts

**선택 이유:**  
매출손익 현황의 월별 막대/선 차트에 필요. React 컴포넌트 방식으로 사용하기 가장 단순하다.

**장점:** React 네이티브, 간단한 API, 반응형 지원  
**단점:** D3 대비 고급 커스터마이징 제한  
**대안:** Chart.js(canvas 기반, React에서 다루기 번거로움), D3(강력하지만 학습 곡선 높음)

### 2-5. 배포 환경 — 기존 VPS (hpa_1.4-g2)

**스펙:** 1 vCPU / 4 GB RAM / NVMe 100 GB / 월 1,000 GB 트래픽  
**비용:** 이미 지불 중 (25,200원/월) — 추가 비용 없음

**왜 Vercel 대신 VPS인가:**
- ERP는 항상 켜진 서버(always-on)가 적합 — Vercel 서버리스는 10초 타임아웃, 콜드스타트 있음
- 수주대장 전체 집계(518건+), Excel 내보내기 같은 무거운 쿼리도 타임아웃 없이 처리
- 이미 비용을 지불 중인 서버를 활용 → 추가 지출 없음
- 대시보드가 거의 미사용 상태 → 여유 리소스 충분

**VPS 배포 구조:**
```
인터넷 → Nginx (80/443) → PM2로 관리되는 Next.js (3000포트)
                        → Supabase (외부 DB, API)
```

**설정 필요 항목:**
- Nginx 리버스 프록시 설정
- Let's Encrypt SSL 인증서 (무료, 자동 갱신)
- PM2로 Next.js 프로세스 관리 (서버 재시작 시 자동 복구)
- GitHub Actions 또는 수동 배포 스크립트

**Supabase 무료 플랜으로 전환:**
- 기존 Pro 플랜($25/월) 불필요 — 매일 사용하므로 2주 미접속 일시정지 없음
- 단, 자동 백업 없음 → VPS에서 직접 pg_dump 백업 크론잡 설정 필요

```bash
# VPS cron 예시 — 매일 새벽 2시 Supabase DB 백업
0 2 * * * pg_dump $DATABASE_URL > /backup/erp_$(date +%Y%m%d).sql
# 30일 이상 된 백업 자동 삭제
find /backup -name "*.sql" -mtime +30 -delete
```

### 2-6. 보안 및 안정성 필수 설정

ERP는 회사 재무 데이터를 다루므로 아래 항목은 선택이 아닌 필수다.

**Supabase RLS (Row Level Security):**
```sql
-- 예시: 수주 테이블 — 로그인한 사용자만 조회 가능
ALTER TABLE 수주 ENABLE ROW LEVEL SECURITY;
CREATE POLICY "인증된 사용자만 접근" ON 수주
  FOR ALL USING (auth.role() = 'authenticated');
```
- 모든 테이블에 RLS 활성화 필수
- anon 키로는 어떤 테이블도 직접 접근 불가하도록 설정
- API 키는 환경변수로만 관리, 코드에 하드코딩 금지

**Supabase Pro 플랜 ($25/월) 필수 이유:**
- 무료 플랜: 2주 미접속 시 DB 일시 정지, 자동 백업 없음
- Pro 플랜: 일별 자동 백업, Point-in-time recovery(7일), 99.9% SLA
- ERP 데이터 유실은 사업 운영에 직결 → 백업은 협상 불가

**TypeScript 필수 이유:**
- AI가 코드 99% 작성 → 타입 오류를 빌드 단계에서 차단
- `수주금액_공급가`에 문자열 입력 시 컴파일 에러로 즉시 발견
- 자동완성으로 AI가 올바른 필드명/타입 사용 강제

**Zod 입력 검증:**
```typescript
// 예시: 투입실적 입력 검증
const 투입실적Schema = z.object({
  수주_id: z.number().positive(),
  투입일: z.string().date(),
  상용직_주: z.number().min(0).max(99),
  // ...
});
// API route에서 DB 저장 전 반드시 검증
const parsed = 투입실적Schema.safeParse(body);
if (!parsed.success) return 400 에러;
```

**감사 추적 (Audit Trail):**
- 모든 핵심 테이블(수주, 투입실적, 기성)에 `생성자`, `생성일`, `수정자`, `수정일` 컬럼 필수
- "누가 언제 바꿨는지" 추적 불가능한 ERP는 분쟁 시 무력

**의존성 최소화 원칙:**
- 패키지 추가 시 "이게 정말 필요한가" 검토
- 유지보수가 중단된 라이브러리는 사용하지 않음
- 핵심 로직(금액 계산)은 외부 라이브러리 없이 순수 TypeScript로 작성

## 3. DB 스키마

### 테이블 목록 (8개)

| 테이블명 | 역할 | 비고 |
|---------|------|------|
| `거래처` | 원청사/발주자 마스터 | 보험료제외율, 하도전용율 보관 |
| `공사단가` | 직종별 주/야간 단가 마스터 | 적용시작일 기준 이력 관리 |

| `계획금액` | 월별 성과 계획금액 | 매월말 공무팀 입력, YYYY-MM 키 |

### 3-1. 거래처 테이블

```sql
CREATE TABLE 거래처 (
  id            SERIAL PRIMARY KEY,
  거래처코드    VARCHAR(10) UNIQUE NOT NULL,  -- P001, S001, S002 ...
  거래처명      VARCHAR(100) NOT NULL,
  보험료제외율  DECIMAL(6,4),   -- 예: 0.075 (7.5%), NULL이면 보험료 없음
  하도전용율    DECIMAL(6,4),   -- 예: 0.85 (85%), 0이면 하도 없음
  법인          VARCHAR(50),
  비고          TEXT,
  생성일        TIMESTAMP DEFAULT NOW()
);
```

**실제 데이터 패턴:**
- 한전: 보험료제외율 NULL, 하도전용율 NULL (직접 발주)
- 정우전설: 보험료제외율 0.075, 하도전용율 0.85
- 에프아이: 보험료제외율 1.0, 하도전용율 1.0 (전액 제외/전액 하도)
- (주)영전사: 보험료제외율 0.075, 하도전용율 여러 값

### 3-2. 공사단가 테이블

```sql
CREATE TABLE 공사단가 (
  id          SERIAL PRIMARY KEY,
  적용시작일  DATE NOT NULL,
  투입구분    VARCHAR(20) NOT NULL,  -- 아래 직종 목록 참고
  주간단가    INTEGER NOT NULL,
  야간단가    INTEGER,
  비고코드    VARCHAR(10),           -- CN170100 등
  생성자      INTEGER REFERENCES 사용자(id),
  생성일      TIMESTAMP DEFAULT NOW(),
```

## 4. 핵심 계산 로직

### 4-1. 수주 금액 계산

```
수주금액_부가세     = 수주금액_공급가 × 0.1
수주금액_합계       = 수주금액_공급가 + 수주금액_부가세
수주금액_보험료제외 = 수주금액_공급가 × (1 - 거래처.보험료제외율)
수주금액_하도적용   = 수주금액_보험료제외 × 거래처.하도전용율
```

> ⚠️ 하도적용은 공급가가 아닌 **보험료제외액**에 하도전용율을 곱함

### 4-2. 투입실적 금액 계산

```
직접노무비 = Σ(직종_주 × 해당직종_주간단가) + Σ(직종_야 × 해당직종_야간단가)
재료비     = (재료비인_주 + 재료비인_야) × 161,000
투입금액   = 직접노무비 + 재료비 + 외주1 + 외주2

일반관리비 = 투입금액 × 0.06   (시스템설정에서 관리)
합계       = 투입금액 + 일반관리비
```

> 단가는 투입일 기준 `적용시작일 <= 투입일` 중 가장 최근 공사단가 레코드 사용  
> 매출손익 현황에서는 **투입금액** 사용 (합계 아님)

### 4-3. 손익 집계

```
누적기성_공급가 = Σ 기성.기성액_공급가 (해당 수주의 전체 기성)
누적기성_부가세 = 누적기성_공급가 × 0.1
누적기성_합계   = 누적기성_공급가 + 누적기성_부가세

잔여금액_하도적용 = 수주금액_하도적용 - 누적기성_합계

성과금액  = 누적기성_공급가  (매출손익 현황에서 사용)
투입금액  = Σ 투입실적.투입금액 (해당 수주의 전체 투입)
순익금액  = 성과금액 - 투입금액
```

## 7. 기존 대시보드(yjsboard.com)와의 관계

### 기존 대시보드 현황

2026년 6월 기준 VPS(hpa_1.4-g2)에서 운영 중인 별도 시스템이 존재한다.

| 항목 | 내용 |
|------|------|
| 도메인 | yjsboard.com (Nginx → 포트 8000) |
| 백엔드 | Python FastAPI (systemd `yjs-dashboard.service`) |
| 프론트엔드 | HTML + Bootstrap 5 + 바닐라 JS |
| DB | SQLite (`schedule.db`) |
| 데이터 소스 | 매일 엑셀 파일 수동 업로드 (`uploads/날짜_백업데이터.xlsx`) |

**현재 데이터 흐름:**
```
EMAX ERP → 엑셀 내보내기 → 담당자 수동 업로드 → FastAPI가 파싱 → 대시보드 화면 표시
```

### 두 시스템을 분리하는 이유

Python/FastAPI/SQLite와 Next.js/Supabase는 런타임 자체가 달라 코드 병합이 불가능하다. 대시보드는 현행 유지하되, ERP는 별도 서브도메인으로 배포한다.

```
yjsboard.com       → 기존 대시보드 (Python FastAPI, 포트 8000)
erp.yjsboard.com   → 새 ERP (Next.js, 포트 3000)
                          ↓
                   Supabase PostgreSQL (공유 DB)
```

### 호환성 설계 원칙

두 시스템을 합치지 않더라도, 새 ERP를 설계할 때 아래 원칙을 지켜 대시보드가 ERP 데이터를 쉽게 읽을 수 있도록 한다.

**1. Supabase REST API를 대시보드의 새 데이터 소스로 활용**

새 ERP가 Supabase에 데이터를 저장하면, 기존 대시보드의 Python 코드에서 엑셀 파일 대신 Supabase REST API를 호출하도록 교체할 수 있다. 엑셀 업로드 단계가 사라지고 실시간 데이터 조회가 가능해진다.

```python
# 기존 대시보드 변경 예시 (FastAPI 내부)
# Before: 엑셀 파일 파싱
# After: Supabase REST API 호출
import httpx
response = httpx.get(
    f"{SUPABASE_URL}/rest/v1/수주",
    headers={"apikey": SUPABASE_ANON_KEY, "Authorization": f"Bearer {SUPABASE_ANON_KEY}"}
)
```

**2. 컬럼명·용어 통일**

대시보드에서 이미 사용 중인 필드명(공사명, 달성율, 기성액 등)을 ERP DB 스키마에서도 동일하게 사용한다. 나중에 대시보드가 API를 직접 호출할 때 파싱 로직 변경을 최소화하기 위함이다.

**3. CORS 설정**

Next.js API Route 및 Supabase에서 yjsboard.com 도메인을 허용 출처(allowed origin)로 등록해두어, 대시보드 프론트엔드에서 직접 Supabase를 호출하는 방식으로 전환할 때도 막힘이 없도록 한다.

**4. Supabase anon key는 읽기 전용 RLS만 허용**

대시보드는 조회만 하면 된다. ERP에서 모든 테이블에 RLS를 설정할 때, `anon` 역할에는 SELECT만 허용하는 정책을 추가해 대시보드가 별도 인증 없이 공개 읽기가 가능하도록 설계한다.

```sql
-- 대시보드용 읽기 전용 정책 예시
CREATE POLICY "대시보드 읽기 허용" ON 수주
  FOR SELECT USING (true);  -- anon도 SELECT 가능
```

> ⚠️ 재무 민감 컬럼(수주금액, 기성액 등)은 별도 정책으로 인증된 사용자만 접근하도록 분리 검토 필요

### 마이그레이션 이후 데이터 흐름 (목표 상태)

```
새 ERP(Supabase) ──REST API──▶ 기존 대시보드 (실시간 데이터)
        │
        └──────────────────▶ erp.yjsboard.com (입력/조회/보고서)
```

엑셀 업로드 단계가 완전히 제거되며, "수동 업로드 프로세스 자동화 제거"를 DX 성과로 보고 가능하다.

### 8-4. 컴포넌트 가이드

**KPI 카드**

레퍼런스 상단의 요약 지표 카드. 대시보드 홈과 매출손익 화면에서 사용.

```
┌──────────────────────────────┐
│  🏗  수주 잔액 (하도적용)      │
│                              │
│  13,650,500,000 원            │
│  ↑ 전월 대비 +12%             │
└──────────────────────────────┘
```

- shadcn `<Card>` 사용
- 아이콘: lucide-react (Building2, TrendingUp, CircleDollarSign 등)
- 수치는 한국식 억/만원 단위로 포맷 (`formatKRW` 유틸 함수 공통 작성)

**데이터 테이블**

수주대장, 투입실적 현황 등 목록 화면에 사용.

- shadcn `<DataTable>` + TanStack Table v8
- 기본 설정: 페이지당 20행, 컬럼 정렬, 텍스트 필터
- 금액 컬럼: 오른쪽 정렬, 천단위 콤마
- 행 클릭 시 상세 슬라이드오버(Sheet) 표시 — 별도 페이지 이동 없이 조회

**폼 (입력 화면)**

투입실적 입력 등 데이터 입력 화면.

- shadcn `<Form>` + react-hook-form + Zod
- 레이블은 폼 요소 위에 배치 (왼쪽 정렬 아님 — 모바일 대응)
- 에러 메시지는 인풋 바로 아래 빨간색 텍스트

**숫자 포맷 유틸 (공통 함수)**

```typescript
// lib/format.ts
export const formatKRW = (n: number) =>
  new Intl.NumberFormat('ko-KR').format(Math.round(n)) + '원';

export const formatEok = (n: number) => {
  if (Math.abs(n) >= 1_0000_0000)
    return (n / 1_0000_0000).toFixed(1) + '억원';
  if (Math.abs(n) >= 10_000)
    return Math.round(n / 10_000) + '만원';
  return formatKRW(n);
};
```

## 9. 데이터 마이그레이션 계획

- `공사단가등록.xlsx` → `공사단가` 테이블

마이그레이션 순서:
1. 거래처 마스터 수동 입력 (23개)

3. 수주 임포트 (518개)
4. 기성 임포트 (수주대장의 1~3차 기성 컬럼 분리)
5. 투입실적 임포트 (약 100개 행 → 실제 입력 레코드로 변환)
